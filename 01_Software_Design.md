# 1과목: 소프트웨어 설계 : 소프트웨어 공학

## 1. 설계 방법론
> 소프트웨어를 어떻게 설계할까? 여러가지 방식 존재

| 방법론 | 설명 | 비유 |
| --- | --- | --- |
| `HIPO` | 시스템을 위계적, 입력-처리-출력 중심 설계 | 회사 조직도처럼 큰 그림 먼저 그리기 |
| V모델 | 개발과 테스트를 동시에 계획 | 레고 만들면서 이상이 있는지 동시에 검수 |
| 나선형 | 반복 개발 + 위험 분석 | 나선형 계단을 돌며 위로 올라가기 |
| 상향식 | 작은 단위(모듈)부터 만들어 점점 큰 시스템으로 | 블록을 하나씩 쌓기 |
| 하향식 | 큰 틀부터 정하고 세부로 내려가는 방식 | 먼저 설계도를 그리고 하나 하나 구현 |
| `Agile(애자일)` | 민첩, 유연 개발 | 팀 프로젝트에서 계속 피드백을 주고 받으며 유연하게 바꾸는 방식 |

### Agile 관련 용어
- Agile 선언문 : 문서보다 사람 간 관계 중심, 적응력 있는 개발
- 종류
  - `SCRUM` : 팀 단위 짧은 주기 반복 개발
  - XP(익스트림 프로그래밍)
    1. 12가지 실천 사항
    2. 핵심 가치 중심 개발

## 2. CASE (Computer-Aided Software Engineering)
- 소프트웨어 개발을 도와주는 자동화 도구
- CASE 도구 : 소프트웨어 설계, 분석, 구현 등을 자동화 도와주는 툴
- CASE 분류 : 설계용 / 분석용 / 구현용
- 비유 : 개발자가 직접 벽돌을 나르지 않고 자동화 기계 사용

## 3. 재공학 (Re-engineering)
- 기존 시스템을 다시 설계해서 더 좋게 만드는 것
- 낡은 소프트웨어를 리모델링하는 느낌

## 4. 시스템 구성 요소

| 구성 요소 | 설명 | 비유 |
| --- | --- | --- |
| 입력(Input) | 데이터를 받는 것 | 주문서 받기 |
| 처리(Process) | 데이터를 처리하는 것 | 요리하기 |
| 출력(Output) | 결과를 내보내는 것 | 음식 내기 |
| 제어(Control) | 순서/흐름을 조절 | 요리 순서 정하기 |
| 피드백(Feedback) | 결과를 보고 다시 반영 | 고객 평가 듣고 맛 조절 |

# 1과목: 소프트웨어 설계 : UML(Unified Modeling Language)

- 소프트웨어 시스템을 시각적으로 표현해 개발자들끼리 같은 그림을 보고 협업할 수 있도록 도와주는 언어

## UML의 분류

1. 행위 다이어그램
2. 구조 다이어그램
3. 럼바우 기법 기반 요소

<br>

### 1. 행위 다이어그램

- 시스템의 동작, 시나리오, 상태 변화를 표현

| 종류 | 설명 | 비유 |
| --- | --- | --- |
| 상호작용 | 객체 간 메시지 주고 받기, 타이밍 | 연극에서 배우들이 대화하는 순서 |
| `유즈케이스` | 사용자와 시스템 간의 상호작용 시나리오 | 사용자가 버튼을 누르면 어떤 일이 벌어지는지 |
| 활동 | 프로세스 흐름도, if-else 조건 처리 포함 | 요리 레시피처럼 순서 따라 행동 |
| 상태 머신 | 객체 상태 변화 중심 | 전구 : 꺼짐 -> 켜짐 상태 전환 |
| 콜라보레이션 | 객체들 간의 협업 구조 | 팀프로젝트에서 누가 어떤 역할을 맡는지 |

<br>

### 2. 구조 다이어그램

- 시스템의 정적인 구조, 즉 부품이 어떻게 구성되어 있는지

| 종류 | 설명 |
| --- | --- |
| 클래스 | 객체 설계도 (속성 + 메서드) / 다양한 관계 표현 : 연관, 의존, 일반화, 집합, 포함, 실체화 등 |
| 복합체 구조 | 클래스 내부 구성까지 보여주는 다이어그램 |
| 배치 | 시스템이 어떤 장비(노드)에 배치될지 표현 |
| 컴포넌트 | 독립적으로 교체 가능한 모듈 표현 | 
| 패키지 | 관련 클래스/ 컴포넌트를 폴더처럼묶은 것 |

#### 클래스의 관계 6종

  - 연관 : 지속적 관계
  - 의존 : 잠깐 쓰는 관계
  - 일반화 : 상속(부모-자식)
  - 집합 : 느슨한 포함(독립 가능)
  - 포함 : 강한 포함 (종속 됨)
  - 실체화 : 인터페이스 구현

<br>

### 3. `럼바우(Rumbaugh)` 기법 기반 요소

- 사람의 기법에서 나온 것으로 객체, 동적, 기능 세 가지 관점으로 시스템을 분석

  - 객체(Object) : 무엇이 존재하는가?
  - 동적(Dynamic) : 어떻게 변화하는가?
  - 기능(Function) : 무엇을 수행하는가?


#### 시험 포인트 요약

| 분류 | 자주 나오는 키워드 |
| --- | --- |
| Use Case | 사용자와 시스템의 상호작용 |
| Activity | 조건 분기, 흐름도 |
| Class | 속성+메서드, 관계 6종(연관~실체화) |
| Deployment | 노드, 장비 설치 위치 |
| Component | 컴파일 단위, 재사용 가능 |

# 1과목 : 소프트웨어 설계 : 요구사항 개발, UI

## 1 요구사항 개발

- 시스템을 만들기 전 무엇이 필요한지, 무엇을 해야하는지 명확하게 정의하는 과정

### SWEBOK 요구개발 프로세스

- `SWEBOK`(Software Engineering Body of Knowledge)에서 정의한 공식적 요구사항 개발 절차

- 비유 : 건물을 짓기 전 설계도 작성

### 요구사항 분석

- 사용자의 요구를 수집하고 정리하는 과정
- 분석 기법 사용 -> 요구사항의 모호함을 줄이고 명확하게
- 비유 : 인터뷰하면서 필요한 것 파악

### 요구사항 분류

| 분류 | 설명 | 
| --- | --- | 
| 기능적 요구사항 | 시스템이 반드시 해야할 기능(예: 로그인, 결제, 검색 기능 등) |
| 비기능적 요구사항 | 성능, 보안, 확장성, 사용성 등 품질과 관련된 요구(예: 3초 이내 응답, 99% 이상 가용성 등) |

- 비유 : 무엇을 할지(기능) + 어떻게 잘할지(품질)를 나누는 것

### 요구사항 명세

- 분석된 요구사항을 문서화, 개발자가 그대로 구현할 수 있도록 구체화
- 비유 : 설계도에서 상세하게 재료와 치수를 적는 것

## 2. UI(User Interface)

- 사용자가 시스템과 직접 상호작용하는 화면, 버튼, 구성 방식 등

### 설계 지침

| 항목 | 설명 | 
| --- | --- |
| 사용자 중심 | 사용자의 입장에서 편리하게 |
| 일관성 | 비슷한 요소는 동일하게 디자인 | 
| 단순성 | 복잡한 기능도 쉽게 표현 |
| 가시성 | 중요한 정보는 눈에 잘 띄게 |
| 표준화 | 디자인 규칙을 통일 | 
| 접근성 | 다양한 사용자(장애인 포함)가 사용 가능하게 |
| 명확성 | 모호한 표현을 피하고 이해하기 쉽게 |
| 오류 방지 | 잘못된 입력은 막고, 실수 시 복구 쉽게 |

- 비유 : 초등학생도 쓸 수 있는 ATM 만들기

### 웹컨텐츠 접근성 지침

- 모든 사람이 정보에 쉽게 접근할 수 있도록 만드는 기준

| 항목 | 설명 |
| --- | --- |
| 인식의 용이성 | 보고, 듣고, 읽기 쉽게 |
| 운용의 용이성 | 조작이 쉬워야 함 |
| 이해의 용이성 | 내용이 잘 이해되어야 함 |

- 비유 : 노약자, 시각장애인도 사용 가능한 웹 만들기

### 설계 원칙

| 항목 | 설명 |
| --- | --- |
| 유효성 | 필요한 정보를 정확히 제공 |
| 직관성 | 별도 설명 없이 사용 가능 |
| 학습성 | 빠르게 익힐 수 있음 |
| 유연성 | 다양한 상황에 맞게 대응 가능 |

- 비유 : 설명서 없이 자연스럽게 조작 가능한 리모컨

# 1과목 : 소프트웨어 설계 : 코드 설계, 구조적 분석 도구

## 1. 코드 설계 

- 프로그램을 작성할 때, 어떤 구조로 어떻게 짤 것인지 고민

### 구조도

- 모듈 간의 호출 관계를 나타낸 도표
- `Fan-in` : 한 모듈을 여러 모듈이 호출 -> 재사용성 좋음
- `Fan-out` : 한 모듈이 여러 모듈을 호출 -> 너무 많으면 복잡해짐
- 비유 : 한 부서에 업무 요청이 몰리는 구조 vs 한 부서가 모든 부서에 명령을 내리는 구조

### 코드 종류

| 분류 기준 | 설명 |
| --- | --- |
| 순차 | 순서대로 명령 수행 |
| `포의숫자` | 반복(Loop) 제어 구조 |
| 블록 | 코드 덩어리 단위 구성 |
| 그룹 분류식 | 같은 성격의 명령끼리 묶음 |

- 비유 : 레시피처럼 순서대로 요리하거나, 반복/그룹화로 편하게 처리

### 코드 오류

| 유형 | 설명 |
| --- | --- |
| `필사` | 코딩 중 단순 오타 |
| `전위` | 실행 순서 잘못 |
| 생략 | 필요한 코드 누락 |
| 임의 | 개발자 마음대로 바꿈 |
| 추가 | 불필요한 코드 삽입 |
| 이중 | 중복된 코드 |

- 비유 : 보고서에 오타 내거나, 순서를 틀리거나, 내용을 두 번 쓰는 실수들

## 2. 구조적 분석 도구

- 소프트웨어를 체계적으로 분석하고 설계하기 위한 도식적 표현 도구

### 자료 흐름도 (DFD: Data Flow Diagram)

| 요소 | 설명 |
| --- | --- |
| 프로세스 | 동그라미(타원) : 처리 기능 |
| 자료 저장소 | 이중선 : 파일 등 저장소 |
| 자료 흐름 | 화살표 : 데이터 이동 경로 |
| `단말` | 사각형 : 외부 요소 (사용자 등) |

- 비유 : 택배시스템
  - 주문서(자료) -> 포장(프로세스) -> 창고(자료 저장소) -> 고객(단말)

### 자료 사전

- 자료 흐름도에서 나오는 모든 데이터의 정의서
- 비유 : 단어장 - 모든 용어를 정의

| 기호 | 의미 |
| --- | --- |
| = | 정의 |
| + | 연결 |
| () | 생략 |
| [] | 선택 |
| {} | 반복 |
| ** | 주석 |

# 1과목 : 소프트웨어 설계 : 모듈, SW 아키텍처

## 1. 모듈 (Module)

- 프로그램을 작은 단위로 쪼개서 개발하는 개념
- 복잡도 낮고 재사용성 높고 유지보수 쉬움

### 설계 방법

- 목적 : 복잡도를 줄이고, 중복을 제거하며 일관성 유지
- 좋은 설계란? 
  - 응집도는 강하게, 결합도는 약하게
- 비유 : 조립식 가구 만들기 - 각 부품은 강하게 연결되되 서로 독립적

### 결합도 - 약할수록 좋음

- 모듈 간 얼마나 서로 의존하는지

| 결합 방식 | 설명 | 
| --- | --- |
| 자료 결합 | 데이터만 주고받음(가장 이상적) |
| `스탬프 결합` | 구조체 등 일부 데이터 공유 |
| 제어 결합 | 제어용 플래그 전달 |
| 외부 결합 | 외부환경(파일, DB 등) 공유 |
| `공통 결합` | 전역 변수 공유 |
| 내용 결합 | 다른 모듈의 내부 접근(최악) |

### 응집도 - 강할수록 좋다 !

- 모듈 내부 요소들이 같은 목적을 위해 얼마나 밀접히 뭉쳐 있는가?

| 응집 유형 | 설명 |
| --- | --- |
| 기능적 | 한 가지 작업만 수행(이상적) |
| 순차적 | 앞 단계 결과를 다음 단계에서 사용 |
| 교환적 | 동일 입력에 따라 다른 처리 |
| 절차적 | 순서 따라 처리만 |
| 시간적 | 시간 맞춰 처리(초기화 등) |
| 논리적 | 유사한 작업 모아둠 |
| 우연적 | 무작위 작업 모음 (최악) |

## 2. SW 아키텍처

- 소프트웨어의 전체 구조와 흐름 설계
- 건축에서의 설계도

### 아키텍처 패턴

| 유형 | 설명 및 비유 |
| 계층 (Layered) | 기능별 층 나누기 - 케이크 위 아래처럼 |
| `MVC`(Model-View-Controller) | 데이터(Model), 화면(View), 제어(Controller) 분리 |
| 클라이언트-서버 | 요청-응답 구조 |
| 파이프-필터 | 데이터 처리 흐름 - 파이프로 이동 |
| Peer-to-Peer | 모든 노드가 동등하게 처리 - 단톡방처럼 서로 요청, 응답 |
| 브로커 | 중간 관리자가 요청 중계 - 고객과 상담원 연결 | 
| 블랙보드 | 공유 저장소에 쓰고 읽음 - 칠판처럼 협업 |

### 품질 속성

- 좋은 SW 아키텍처의 조건들

| 속성 | 설명 |
| --- | --- |
| 성능 | 빠른 응답 |
| 사용 용이성 | 쓰기 쉬움 |
| 보안성 | 외부 공격 방지 |
| 시험 용이성 | 테스트하기 쉬움 |
| 가용성 | 멈추지 않고 계속 작동 |
| 변경 용이성 | 구조 바꿔도 쉽게 대응 |
| 사용성 | 사용자 친화적 |

- 비유 : 건물 설계 시 내진설계, 접근성, 확장성 모두 고려

### 프레임워크

- 소프트웨어의 기본 구조와 뼈대
- 재사용 가능하고 확장 가능 구조
- 비유 : 레고판 - 거기에 새로운 블록만 붙이면 됨
